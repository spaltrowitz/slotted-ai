## Findings Summary

**Severity Breakdown:** 3 Critical, 3 High, 4 Medium (Accessibility), 0 Low

### Critical Issues
1. **Hardcoded email in production** — `localStorage.setItem('slotted_referrer_email', 'sharipaltrowitz@gmail.com')` at `AuthContext.tsx:65`; personal PII exposed
2. **Sensitive console logs of credentials** — Apple Calendar username, full API responses, FCM tokens logged in AuthContext and usePushNotifications
3. **Firebase SW placeholder keys** — TODO placeholder API keys in `public/firebase-messaging-sw.js`; push notifications non-functional in production

### High Issues
1. **Open redirect in OAuth flows** — `AuthContext.tsx:236, 305` trusts server response without domain validation; `window.location.href = data.url` allows attacker-controlled redirects
2. **Direct fetch() bypasses token interceptor** — `AuthContext.tsx:122-140` uses raw `fetch()` instead of `api` client, missing token refresh logic
3. **Performance: FriendsPage creates functions per render** — `renderFriendRow` creates new handlers on every render; needs React.memo extraction

### Accessibility Gaps
- StarRating: no ARIA roles, no keyboard navigation
- AddToCalendarModal: no `role="dialog"`, no focus trap
- CalendarPicker: checkboxes lack `<label>` elements
- FriendsPage: `role="button"` elements lack full keyboard support

### TypeScript Debt
12+ instances of `err: any` in catch blocks; should be typed as `AxiosError` or `Error`.

## Security Positives

✅ Auth tokens: Fresh `getIdToken()` per request via axios interceptor  
✅ No XSS: No `dangerouslySetInnerHTML` anywhere  
✅ Route protection: ProtectedRoute properly guards authenticated pages  
✅ CSRF: Bearer token auth inherently CSRF-resistant  
✅ Environment vars: Firebase config uses `VITE_` prefix correctly  
✅ PWA caching: NetworkOnly for auth endpoints  
✅ Code splitting: All routes lazy-loaded with retry logic
