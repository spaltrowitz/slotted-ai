## Critical Finding

**Groups feature is broken in production:** Frontend `GroupAvailability.tsx:52` calls `POST /availability/group-overlap` — this endpoint DOES NOT EXIST. Backend only has `POST /availability/multi-friend-overlap`. Users selecting multiple friends to find common time get immediate 404 error. Full group CRUD missing (schema exists, zero API endpoints for create/list/update/delete).

## High-Priority Issues

1. **Simultaneous cross-friend-requests** create permanent pending state — upsert overwrites invited_by, trapping one user
2. **No future-time validation** on meetup creation — past meetups accepted and pollute list
3. **No endTime > startTime validation** — backward meetups created silently
4. **`zonedToUtc` DST edge-case** — off-by-hour during transitions (Intl.DateTimeFormat + reference date bug)
5. **No CRUD endpoints for `friend_groups` table** — schema exists, zero API
6. **Counter-propose doesn't auto-cancel** original 1:1 meetup for proposer

## Medium-Priority Issues

- `syncUserToDb` silent failure (raw fetch bypass) → user stuck in 404 loop
- Onboarding race: completeOnboarding called even if mutation fails

- Upsert on re-request resets other user's friendship_type

## Key Architectural Insights

- **Notification type overload:** `meetup_request` used for 4+ semantic events causes false dedup suppression
- **Calendar sync destructive:** DELETE all → INSERT new creates brief zero-availability window
- **Group schema orphaned:** Tables exist (friend_groups, friend_group_members) but no API layer (CRUD missing entirely)
- **Time validation missing:** No temporal checks on meetup creation (past times, backward times both accepted)
- **Timezone fragility:** Calendar sync uses server locale for notification text, not per-recipient TZ
